# 对话历史记忆功能 - 设计文档

项目：Buddy | 日期：2026-04-06

## 需求概述

为现有的视频对话系统添加**对话记忆功能**：
- 用户可以查看历史对话记录
- 用户可以点击历史对话继续对话
- 用户可以新建对话清空当前内容
- 用户可以删除不需要的历史对话
- 支持存储对话向量，用于后续RAG检索增强

## 前端设计

### 整体布局

- **位置**："历史记录"按钮放在**右上角**，在"退出登录"按钮左侧，和其他按钮保持一致样式
- **交互**：点击按钮展开下拉菜单，再次点击或点击外部收起
- **下拉菜单尺寸**：宽度 320px，最大高度 450px，超出滚动

### 下拉菜单结构

从上到下：
1. **当前对话**（高亮显示）
   - 显示当前对话标题（取第一条消息前30个字）
   - 不显示删除按钮（需求确定：不需要删除当前对话，新建对话即可）
2. **分隔线**
3. **历史对话列表**
   - 按时间倒序排列（最新的在最上面）
   - 每条显示：时间 + 对话标题预览
   - 每条右侧显示删除图标（🗑），点击弹出确认框确认删除
   - 点击任意历史对话，加载该对话的所有消息到当前会话
4. **分隔线**
5. **新建对话按钮**
   - 绿色按钮 + "+ 新建对话" 文字
   - 点击后：如果当前对话为空不保存，直接清空；否则保存当前对话到历史，清空当前开始新对话

### 交互细节

- **删除**：点击删除图标 → 弹出确认对话框 "确定要删除这条对话记录吗？" → 确认后删除并刷新列表
- **切换对话**：点击历史项 → 保存当前对话 → 加载选中对话的所有消息到当前界面
- **新建对话**：点击按钮 → 保存当前对话（如果有消息）→ 创建新对话并清空当前界面

## 后端设计

### 存储架构

当前后端使用 CSV 文件存储，延续这个架构，配合 ChromaDB 存储向量：

- **对话元信息 + 消息内容**：CSV 文件存储（延续现有架构）
- **向量嵌入**：ChromaDB 嵌入式向量数据库，文件存储

### CSV 文件设计

#### 1. `conversations.csv` - 对话元信息

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 主键 |
| `user_id` | String | 用户ID（匹配现有UserSession） |
| `title` | String | 对话标题（取第一条用户消息前30字） |
| `created_at` | ISO 格式字符串 | 创建时间 |
| `updated_at` | ISO 格式字符串 | 最后更新时间 |
| `is_active` | Boolean | 是否是当前激活对话 |

#### 2. `messages.csv` - 消息内容

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 主键 |
| `conversation_id` | UUID | 所属对话ID |
| `role` | String | 角色：`user` / `model` |
| `content` | Text | 消息内容 |
| `created_at` | ISO 格式字符串 | 创建时间 |

### 向量存储 - ChromaDB

- 集合（Collection）命名：`conversation_messages`
- 每条消息对应一条Chroma记录：
  - `id`: message id (UUID)
  - `embedding`: 生成的向量
  - `metadata`: `{"conversation_id": "...", "role": "user/model", "created_at": "..."}`
  - `document`: 消息内容

ChromaDB 数据持久化到 `backend/data/chroma/` 目录，和 CSV 文件放在一起。

### API 接口设计

所有接口需要认证（依赖现有认证体系）：

| 方法 | 路径 | 功能 | 返回 |
|------|------|------|------|
| `GET` | `/api/conversations` | 获取当前用户所有对话列表 | `[{id, title, updated_at, is_active}, ...]` |
| `GET` | `/api/conversations/{id}` | 获取指定对话的完整消息列表 | `{id, title, messages: [{role, content, created_at}, ...]}` |
| `POST` | `/api/conversations` | 创建新对话 | `{id, title}` |
| `POST` | `/api/conversations/{id}/messages` | 添加消息到对话（同时计算向量） | `{success}` |
| `PUT` | `/api/conversations/{id}` | 更新对话信息（标题等） | `{success}` |
| `DELETE` | `/api/conversations/{id}` | 删除对话（级联删除所有消息和向量） | `{success}` |

### 向量生成流程

当新消息添加到对话时：
1. 后端收到用户消息
2. 保存消息到 `messages.csv`
3. 调用嵌入模型（OpenAI 或 Anthropic，复用现有LLM配置）生成向量
4. 将向量存入 ChromaDB，附带元数据
5. 更新对话 `updated_at` 时间
6. 返回成功

这样后续需要检索相关历史时，可以基于向量相似度查找相关对话片段。

## 前端状态管理

### 当前实现调整

现有 `VideoCallPage.vue` 中 `conversationMessages` 是本地 `ref`，需要：

1. 创建对话后，保存 `currentConversationId` 状态
2. 每次添加新消息时，除了本地更新，同时调用后端API保存到数据库
3. 切换对话时，调用API加载完整消息列表替换本地状态
4. 新建对话时，先保存当前对话（如果有消息），然后创建新对话清空本地

### 数据流向

```
用户点击历史对话
  → 请求 GET /api/conversations/{id}
  → 更新 conversationMessages.value = 返回的 messages
  → 更新 currentConversationId

用户发送新消息
  → 追加到 conversationMessages.value
  → 调用 POST /api/conversations/{id}/messages 保存到后端
  → （后端自动生成向量）

用户点击新建对话
  → 如果当前对话有消息，保存当前
  → 调用 POST /api/conversations 创建新对话
  → 清空 conversationMessages.value
  → 更新 currentConversationId = 新对话id

用户点击删除对话
  → 弹出确认
  → 调用 DELETE /api/conversations/{id}
  → 从列表中移除，刷新下拉菜单
  → 如果删除的是当前对话，自动创建新对话清空当前
```

## 功能边界说明

**本次实现范围：**
- ✅ 完整的对话CRUD
- ✅ 对话列表下拉菜单
- ✅ 切换对话、新建对话
- ✅ 删除对话
- ✅ 存储向量用于RAG
- ✅ 每次对话时，完整对话历史都会发给模型（用户需求）

**不包含在本次实现：**
- ❌ 对话重命名功能
- ❌ 搜索历史对话功能（可以后续加）
- ❌ 批量删除/清空全部
- ❌ 移动端左滑优化（后续优化）
- ❌ 向量检索（只存储，检索功能后续实现）

## 风险和考虑

1. **令牌消耗**：对话越长，发给模型的token越多。当前需求就是要把全量历史发给模型，所以接受这个 tradeoff
2. **向量生成成本**：每条消息都生成向量，会增加调用成本和延迟，可以后续考虑异步生成优化
3. **下拉菜单性能**：历史对话很多时（几百条），下拉滚动会有点卡，可以后续加分页或无限加载
