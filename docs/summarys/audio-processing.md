# 音频处理与语音识别开发总结

## 概述

Buddy 使用阿里云实时语音识别（ASR）服务进行流式语音识别，配合静音检测自动分段，用户说完一句话后自动触发LLM回复，无需手动点击发送，体验流畅。

---

## 关键问题解决

### 阿里云实时语音识别 空闲超时修复

**问题现象：**
```
Gateway:IDLE_TIMEOUT:Websocket session is idle for too long time, the last directive is 'StartTranscription'!
```

**根本原因：**  
原分步流程：用户先点「开始连接」，再点「开始录音」→ 连接建立后到发送音频之间有空闲 → 阿里云超时断开。

**解决方案：** 调整执行顺序，完全消除空闲时间
```
用户点击「开始录音识别」
   ↓
1. 先请求麦克风权限 → 用户授权完成后代码才继续
   ↓
2. 麦克风授权成功 → 建立 WebSocket 连接
   ↓
3. 连接成功 → 立即发送 StartTranscription 启动识别
   ↓
4. 立即开始音频采集并持续发送音频数据
```

这样间隔 < 100ms，从根源解决了IDLE_TIMEOUT问题。

## 核心组件实现

### `AsrStreamProcessor` 实现

创建了独立处理器：
- 音频流接收与处理
- 静音超时检测（默认1500ms）
- 部分结果与最终结果回调
- 文本合并与状态管理
- 完全异步设计

---

### 流式ASR效率评审

**发现的问题：**
| 问题 | 严重程度 |
|------|----------|
| 每个消息都在WARNING级别记录完整数据 → 生产日志过大 | 中等 |
| 快速重复开关可能产生未引用异步任务 | 低 |
| 几个不必要的存在性检查 | 低 |

**总体评估：** 实现设计高效，适合实时流式ASR：
- 正确的双缓冲解决连接建立竞争
- 正确的资源清理和边界检查
- 全程异步，热路径没有阻塞调用
- 职责分离清晰

## 架构设计

### 统一语音架构 (重构后)

遵循和LLM相同的统一架构模式：
```
backend/services/speech/
├── base.py           # BaseASRService, BaseTTSService 抽象基类
├── factory.py        # ASR/TTS 统一工厂
├── asr_stream_processor.py  # ASR 流式处理器
└── providers/        # 具体实现
    ├── aliyun_token.py
    ├── aliyun_asr.py
    ├── aliyun_tts.py
    └── aliyun_streaming_asr.py
```

**优点：**
- 添加新提供者只需要实现新类，不修改上层代码
- 工厂根据配置自动选择，符合开闭原则
- 上层只依赖抽象接口，不关心具体厂商

### 静音检测分段设计

**问题：** 用户说话会有停顿，如何判断一句话说完了？

**解决方案：** 静音超时方案
- 用户每次说出新内容，重置一个1.5秒定时器
- 如果1.5秒没有新语音，判定为句子结束，触发LLM回复
- 实现简单，不需要训练模型，实际体验良好

---

## 前端体验优化

- **蹦字效果**：ASR部分结果实时更新，用户看到识别进度，减少等待感
- **定时器重置**：每次新语音重置静音定时器，正确处理自然停顿
- **日志采样**：前端音频每10个chunk记录一次，避免控制台刷屏

---

## 经验总结

1. **协议细节很重要** - 第三方WebSocket协议对超时的限制可能超出预期，需要仔细分析问题根源
2. **简单方案往往最好** - 静音检测使用超时方案比ML模型简单，实际效果满足需求
3. **统一架构模式降低认知负担** - LLM和语音都使用相同的抽象+工厂+提供者模式，容易理解和维护
4. **效率评审提前发现问题** - 在开发早期进行效率评审，避免后期重构
