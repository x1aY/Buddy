# 阿里云实时语音识别 demo 修复总结

## 问题
运行 `docs/refs/实时语音识别前端demo.html` 时报错：
```
Gateway:IDLE_TIMEOUT:Websocket session is idle for too long time, the last directive is 'StartTranscription'!
```

## 根本原因分析
原代码采用两步操作流程：
1. 用户点击「开始连接」→ 建立 WebSocket → **立即发送 `StartTranscription`**
2. 用户点击「开始录音」→ 开始发送音频数据

问题：在用户点击「开始连接」到点击「开始录音」之间，连接处于空闲状态，没有任何数据发送。阿里云实时语音识别服务会因为空闲超时断开连接。

进一步测试发现：即使将 `StartTranscription` 移到「开始录音」阶段，如果用户点击「开始连接」后停留几秒再点击「开始录音」，整个 WebSocket 连接还是会超时断开（阿里云对整个连接都有空闲超时限制）。

## 解决方案
采用**一步操作流程**，并调整执行顺序消除所有空闲时间：

```
用户点击「开始录音识别」
  ↓
1. 先请求麦克风权限 → 用户授权完成后代码才继续
  ↓
2. 麦克风授权成功 → 建立 WebSocket 连接
  ↓
3. 连接成功 → 立即发送 StartTranscription 启动识别
  ↓
4. 立即开始音频采集并持续发送音频数据
```

这样从连接建立到开始发送音频数据间隔 < 100ms，**完全消除了空闲时间**，从根源上解决了 IDLE_TIMEOUT 问题。

## 修改内容
文件：`docs/refs/实时语音识别前端demo.html`

1. **调整按钮状态**：页面加载完成后「开始录音识别」直接可用，无需先连接
2. **调整执行顺序**：先获取麦克风权限，再建立 WebSocket 连接
3. **合并操作流程**：用户单次点击完成所有初始化，无需分步操作
4. **保留手动连接**：保留了「开始连接（手动）」「断开连接」按钮供手动控制使用

## 验证结果
- ✅ 不再出现 IDLE_TIMEOUT 错误
- ✅ WebSocket 连接稳定，持续传输音频数据
- ✅ 阿里云服务正常返回识别结果（SentenceBegin → TranscriptionResultChanged → SentenceEnd）
- ✅ 实时语音识别功能正常工作

## 经验总结
阿里云实时语音识别 WebSocket 协议对空闲连接超时限制比较严格（大约几秒）。任何时候连接建立后都必须尽快发送 `StartTranscription` 和音频数据，不能有较长空闲。将用户授权操作放在连接建立之前，可以保证连接建立后立即开始数据传输。
