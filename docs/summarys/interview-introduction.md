# Buddy 项目面试介绍

> 整理日期：2026-04-13

## 项目简介

这是一个**基于大语言模型的多模态实时对话系统**，我给它起名叫 Buddy。用户可以通过摄像头和麦克风与AI面对面交流，AI能够"看到"你展示的东西，听懂你说的话，并且可以联网搜索实时信息，用语音回复你。这本质上是一个移动端AI助手的Web原型。

---

## 项目亮点

### 1. **现代全栈架构设计**
- 前端采用 **Vue 3 + TypeScript + TailwindCSS**，响应式设计，Vite构建，开发体验好
- 后端采用 **FastAPI + WebSocket**，天生支持异步，原生类型安全，适合AI流式场景
- 通过WebSocket实现全双工实时通信，音频、视频帧都走一条连接，延迟低

### 2. **统一LLM架构，多提供商支持**
- 设计了抽象基类 `BaseLLMService`，定义统一接口
- 目前无缝支持 **Anthropic Claude、OpenAI、火山引擎豆包** 三种提供商
- 使用工厂模式根据配置动态选择，想加新供应商只需要写一个适配层
- 支持多模态输入，可同时处理文本和摄像头图像

### 3. **完整的工具调用闭环**
- 实现了标准的 **ReAct-style 工具调用循环**，最大支持5轮迭代
- 内置了 open-websearch 联网搜索工具，LLM可以查询实时信息
- 从工具定义、调用解析、执行到把结果放回LLM，整个流程闭环
- 这个设计让AI能回答超出它训练数据的问题，比如查最新天气、新闻

### 4. **流式语音处理，用户体验优秀**
- ASR 流式识别，前端做了**蹦字效果**，用户可以实时看到识别进度
- 智能**静音检测分段**：检测到1.5秒静音自动判断为一句话结束，触发LLM回复，用户不需要手动点击发送
- LLM回复也是流式输出，配合TTS可以边生成边播放，减少等待感

### 5. **现代化开发流程**
- 用 **Figma** 做前端页面设计，确定布局和交互流程再编码
- 用 **superpowers (SDD - Situational Driven Development)** 做软件开发，先想清楚场景和需求再写代码
- 用 **Chrome DevTools MCP** 做网页自动化测试，开发时直接在浏览器调试

## 遇到的难点与解决方案

### 1. **异步死锁问题 - 语音触发后LLM永远不回复**
**问题**：语音识别完成后回调触发LLM生成，但两个函数都需要获取同一个会话锁，导致嵌套获取锁产生死锁，进程卡住，没有任何日志输出，很难定位。

**解决**：
- 重新梳理锁的作用范围，移除了嵌套获取锁的设计
- 添加了完整的顶层异常捕获，避免静默失败
- **经验教训**：异步编程中一定要谨慎使用锁，能不用就不用，一定要想清楚调用链

### 2. **多LLM提供商协议不兼容**
**问题**：不同厂商的API协议差异很大：
- 消息角色定义不一样：Anthropic 要求工具结果是 `tool` 角色，但火山引擎不支持这个角色
- 认证方式不一样：有的用 `x-api-key`，有的用 `Authorization: Bearer`
- 工具调用格式也不统一

**解决**：我采用了**适配层设计**：
- 对外只暴露统一接口，参数是我们自己定义的标准消息格式
- 每个提供商在自己的适配类里做格式转换，比如把`tool`角色转成`user`角色
- 符合开闭原则，新增提供商不影响现有代码

### 3. **ASR何时分段触发LLM这个问题很微妙**
**问题**：用户说话会有停顿，不能识别一个字就触发一次，也不能等用户完全结束。怎么判断一句话说完了？

**解决**：采用**静音超时方案**：
- 用户每次说出新内容，我就重置一个1.5秒的定时器
- 如果1.5秒没有新语音，判定为句子结束，触发LLM回复
- 这个方案简单，不需要训练模型，实际用起来体验很好，大多数情况都能正确分段

### 4. **工具调用循环设计**
**问题**：LLM调用工具后，需要把工具结果返回给LLM，LLM基于结果再回答。这个循环怎么设计清晰？

**解决**：
- 抽取出独立的 `LlmPipeline` 类管理整个流程
- 维护一个 `working_messages` 列表保存当前轮次的中间消息（工具调用和结果）
- 循环直到LLM不再调用工具，或者达到最大迭代次数（防止无限循环）
- 结构清晰，每个步骤职责单一，好调试好扩展

### 5. **流式前端状态同步**
**问题**：ASR partial结果、ASR final结果、LLM流式输出、TTS播放，多个异步事件同时发生，前端状态容易乱。

**解决**：
- 每个消息都带唯一的 `segment_id`，前端按ID分组更新
- 增量更新DOM，不用全量重绘，性能更好
- 使用Vue响应式自动同步，状态变更视图自动更新

## 个人收获

这个项目让我对**AI原生应用开发**有了更深的理解：
- AI应用和传统CRUD应用架构不一样，流式、异步、多轮是常态
- 好的抽象设计很重要，LLM提供商一直在变，但核心流程不变
- 用户体验细节决定成败，一点点延迟感受差很多，流式处理是必须的
- 调试异步问题要靠日志和分层设计，好的日志能帮你快速定位问题
